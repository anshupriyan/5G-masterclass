# 05. 5G Unified Data Management (UDM) and Unified Data Repository (UDR)

The 5G Core (5GC) introduces a decoupled, stateless network design for subscriber data management, separating database storage from front-end processing logic. This is achieved via two specialized Network Functions: **Unified Data Management (UDM)** and the **Unified Data Repository (UDR)**.

![[Materials/Slides/UDM_Slide_1.png]]

---

## ☁️ 1. Stateless Cloud-Native Design

In traditional 3G and 4G networks, subscriber data and application logic were bundled together in a stateful database (like the HLR/HSS). If a database node crashed, both active session logic and stored profiles were compromised.

![[Materials/Slides/UDM_Slide_2.png]]

5G SBA completely decouples this architecture to create a stateless cloud design:
* **Front-End (Application Logic):** Handled by state-free, lightweight application engines (like the UDM). If a UDM instance fails, another instance can instantly spin up and process requests since it holds no internal subscriber states.
* **Back-End (Data Storage):** Handled by highly resilient, centralized databases (like the UDR).
* **UDSF (Unstructured Data Storage Function):** In addition to the structured UDR, 5G introduces the UDSF. Stateless NFs (like the AMF or SMF) can dynamically store and retrieve their transient operational session states (e.g. active call variables) inside the UDSF, ensuring zero active session loss if an NF instance crashes.

## 🗄️ 2. The Unified Data Repository (UDR)

The **Unified Data Repository (UDR)** is the primary physical database of the 5G Core.

![[Materials/Slides/UDM_Slide_3.png]]

### Key Mandates:
* **Centralized Storage:** Acts as the master storage repository for all structural data within the core network.
* **Data Collections Stored in UDR:**
  1. **Subscription Data:** Subscriber profile records, IMSI/SUPI configurations, and authentication credentials (e.g., subscription key $K$, ciphering keys, and network authentication algorithms).
  2. **Policy Data:** Static policy templates, subscription-based limits, and PCC (Policy and Charging Control) rules.
  3. **Structured Data for Exposure:** Data records exposed to external third-party Application Functions (AFs).
  4. **Application Data:** Transient state data saved by active applications.

---

## 🧠 3. The Unified Data Management (UDM)

The **Unified Data Management (UDM)** is the state-free front-end application processor that retrieves data from the UDR to execute control plane operations.

![[Materials/Slides/UDM_Slide_4.png]]

### Key Mandates:
* **Decoupled Architecture:** Acts as the front-end logic engine for subscription data stored in the UDR.
* **Subscriber Services Handled by UDM:**
  * **Access Management:** Verifies subscriber roaming agreements, home PLMN configurations, and checks if a user has access rights to specific network slices (S-NSSAI) and DNNs.
  * **Registration Management:** Updates and tracks which AMF currently services the active UE. When a UE registers with a new AMF, the UDM records the new location and sends an explicit **Location Cancellation** command to the old AMF to purge legacy contexts.
  * **Authentication:** Generates the raw authentication vectors and security credentials (generating RAND, AUTN, and the anchor key $K_{\text{ASME}}$ or $K_{\text{SEAF}}$) required by the AUSF (via the N13 interface) to execute mutual authentication challenges.
  * **SMS Management:** Coordinates over-the-air SMS routing and delivery.
