# 05. 5G MAC Sublayer Architecture

The **Medium Access Control (MAC)** layer is the workhorse of the lower Layer 2 protocol stack. It sits directly beneath the **RLC** layer and directly above the **Physical (PHY)** Layer. It takes **RLC PDUs** as its **MAC SDUs** and packages them into **MAC PDUs** (commonly known as **Transport Blocks**) to pass to the physical radio hardware.

![[MAC_Slide_1.png]]

---

## 🗺️ The Four Foundational Pillars of the MAC Layer

According to the 3GPP specifications, the MAC sublayer is responsible for four massive network mandates:

![[MAC_Slide_2.png]]

1. **Logical-Channel Multiplexing and Demultiplexing**
2. **Scheduling**
3. **Hybrid-ARQ (HARQ) Retransmissions**
4. **Multiplexing/Demultiplexing for Carrier Aggregation (CA)**

---

### 🔀 Pillar 1: Channel Mapping & Multiplexing

The MAC layer serves as the ultimate highway junction of the radio interface, defining boundaries by how data is categorized:

* **Logical Channels (Services to RLC):** Defined by the **type of information** being carried. It splits into *Control Channels* (system management) and *Traffic Channels* (user data).
* **Transport Channels (Services from PHY):** Defined by **how and with what characteristics** the data is physically blasted across the radio spectrum (e.g., paging, broadcasting, or shared streams).

![[MAC_Slide_3.png]]

The MAC layer acts as the master mapping matrix that weaves these pipelines together:

![[MAC_Slide_4.png]]

#### The Comprehensive Channel Inventory

| Channel Type | Acronym | Full Name | Operational Domain & Purpose | Maps To (via MAC) |
| :--- | :--- | :--- | :--- | :--- |
| **Logical** | **BCCH** | Broadcast Control Channel | **Control:** Beams downlink system information parameters (MIB/SIBs) to all devices in the cell. | **BCH** (or DL-SCH) |
| **Logical** | **PCCH** | Paging Control Channel | **Control:** Alerts devices whose exact cell-level location is unknown when incoming data arrives. | **PCH** |
| **Logical** | **CCCH** | Common Control Channel | **Control:** Used by devices executing initial random access connection setup before they have a dedicated ID. | **DL-SCH** / **UL-SCH** |
| **Logical** | **DCCH** | Dedicated Control Channel | **Control:** A custom, point-to-point path carrying explicit RRC configuration commands to an active phone. | **DL-SCH** / **UL-SCH** |
| **Logical** | **DTCH** | Dedicated Traffic Channel | **Traffic:** Dedicated point-to-point channel carrying actual internet, web, or voice payloads. | **DL-SCH** / **UL-SCH** |
| | | | | |
| **Transport** | **BCH** | Broadcast Channel | **Downlink:** Has a fixed, pre-defined transport format to guarantee any device can decode it instantly. | *Receives BCCH* |
| **Transport** | **PCH** | Paging Channel | **Downlink:** Supports UE sleep cycles via Discontinuous Reception (DRX) to save device battery. | *Receives PCCH* |
| **Transport** | **DL-SCH** | Downlink Shared Channel | **Downlink:** The heavy-lifting data pipe supporting dynamic rate adaptation, beamforming, and HARQ. | *Receives CCCH, DCCH, DTCH* |
| **Transport** | **UL-SCH** | Uplink Shared Channel | **Uplink:** The primary uplink channel for phone transmissions, capable of dynamic scheduling and power adjustment. | *Receives CCCH, DCCH, DTCH* |
| **Transport** | **RACH** | Random-Access Channel | **Uplink:** A thin contention channel where phones blast brief preamble bursts to get the network’s initial attention. | *Bypasses logical channels* |

### ⏱️ Pillar 2: Dynamic Scheduling

The gNodeB base station houses a centralized **MAC Scheduler**. Because the radio waves are a shared environment, no phone can transmit or receive without the MAC scheduler’s permission.

* It continuously samples the cell's environment, checking things like device signal-to-noise ratios (CQI - Channel Quality Indicator), data priorities (QoS parameters), and buffer status reports.
* It dynamically commands exactly which subcarriers and time slots are granted to specific users, down to a millisecond or fractional-millisecond level.

---

### 🧬 Pillar 3: MAC Control Elements (MAC CEs)

A unique feature of the MAC layer is its ability to inject its own network-layer commands directly into the data payload using **MAC Control Elements (MAC CEs)**.

Instead of wasting time sending a massive, slow Layer 3 control message (like an RRC command) to handle routine, urgent operational adjustments, the MAC layer slips micro-commands directly into the data frame:

![[MAC_Slide_5.png]]

#### Inside a MAC PDU Frame
A finished MAC PDU consists of several **MAC SDUs** (your actual internet data), **MAC Control Elements (MAC CEs)**, and a set of accompanying **Headers (H)** summarizing the layout:

```text
+----+-----------+----+-----------+----+----------+---------+
| H  |  MAC SDU  | H  |  MAC SDU  | H  |  MAC CE  | Padding |
+----+-----------+----+-----------+----+----------+---------+
                                       ^
                        [ Injected MAC Layer Commands ]
```

#### Primary Functions Managed by MAC CEs:
* **Scheduling Info / Buffer Status Reports (BSR):** The phone uses a MAC CE to tell the base station: *"Hey, my internal buffer is holding 5MB of data, I need a larger upload slot."*
* **Random Access Parameters:** Handles coordination flags during initial network entry routines.
* **Timing Advance (TA):** If you are driving away from a cell tower at 100 km/h, your signal takes a few microseconds longer to hit the tower. The base station’s MAC layer tracks this delay and fires a **Timing Advance MAC CE** telling your phone: *"Shift your internal transmission clock forward by 3 microseconds so your signal lands perfectly in sync with everyone else's."*

---

### ⛓️ Pillar 4: Carrier Aggregation (CA) Multi-Stack Binding

When a network activates **Carrier Aggregation (CA)** to boost your download speeds, it binds multiple different radio frequency bands together simultaneously (e.g., combining a wide 700MHz coverage band with a blazing fast 3.5GHz capacity band).

The MAC layer acts as the master split point for Carrier Aggregation:

![[MAC_Slide_6.png]]

* Above the MAC layer (SDAP, PDCP, RLC), there is only **one single architecture thread** processing your application data.
* When the data hits the MAC layer, it splits the single data stream, multiplexing and demultiplexing blocks across multiple parallel **Physical Layer (PHY)** channel components simultaneously (one per connected cell component).

## 🔁 The Hybrid-ARQ (HARQ) Protocol Engine

While the higher RLC layer manages software-driven ARQ retransmissions, the MAC layer runs a blindingly fast, hardware-accelerated feedback loop called **HARQ (Hybrid Automatic Repeat Request)** to catch errors almost instantly.

### The Stop-and-Wait Multi-Process Pipeline
If the network sent a packet and had to pause until the receiver sent a confirmation back before sending the next one, the entire connection would grind to a halt.

![[MAC_Slide_7.png]]

To keep data flowing smoothly, 5G uses a **Stop-and-Wait Multi-Process Protocol**:
* The MAC layer maintains multiple independent, parallel HARQ processing engines (e.g., Process 0, Process 1, Process 2, Process 3) running simultaneously.
* While Process 0 is pausing and waiting for its over-the-air feedback delivery confirmation, the transmitter can instantly pump out data using Process 1, Process 2, and Process 3 right behind it.

### The Core Shift: 4G (Synchronous) vs. 5G (Asynchronous)
* **4G LTE utilized Synchronous HARQ:** Transmissions and retransmissions were locked to strict, pre-calculated time slots. If an error happened, the retransmission could *only* happen at a specific microsecond index later.
* **5G NR utilizes Asynchronous HARQ:** Retransmissions can be scheduled to happen at any time on any available process slot.

![[MAC_Slide_8.png]]

* Retransmission parameters and process identities are communicated out-of-band via **DCI (Downlink Control Information)** parameters carried on the **PDCCH (Physical Downlink Control Channel)**. This provides the network with incredible scheduling flexibility, helping it achieve ultra-low latency.

## 🏗️ CBG (Code Block Group) Retransmissions: The Ultimate 5G Efficiency Feature

In 5G, when a massive piece of data is prepared for transmission, it is called a **Transport Block (TB)**.

![[MAC_Slide_9.png]]

```text
[   MASSIVE 5G TRANSPORT BLOCK (TB)   ]
                   ↓  (Channel Coding Split)
[ CB 0 ] [ CB 1 ] [ CB 2 ] [ CB 3 ] [ CB 4 ] [ CB 5 ] [ CB 6 ] [ CB 7 ]
\_______________/ \_______________/ \_______________/ \_______________/
        ↓                 ↓                 ↓                 ↓
    [ CBG 0 ]         [ CBG 1 ]         [ CBG 2 ]         [ CBG 3 ]
                      (Error Here!)
                          │
                          ▼
             [ Retransmit ONLY CBG 1 ]
```

### The Old 4G Problem
During the physical layer channel coding phase, a huge Transport Block is divided into smaller chunks called **Code Blocks (CBs)**, and each chunk gets its own CRC error check.
In 4G LTE, if even a single Code Block was corrupted by interference, the network had to throw away the *entire massive Transport Block* and retransmit the whole thing. This was highly inefficient and wasted a lot of bandwidth.

### The 5G Solution: Code Block Groups (CBG)
5G NR solves this by introducing **Code Block Groups (CBG)**:
1. Individual Code Blocks are bundled together into logical sets called **Code Block Groups (CBGs)**.
2. If an over-the-air error occurs, the receiver identifies the specific corrupted Code Block.
3. Instead of requesting a retransmission of the entire Transport Block, the receiver's feedback explicitly calls out the affected **Code Block Group** (via a CBG-based HARQ ACK/NACK bitmap).
4. The base station retransmits **only that single specific Code Block Group**.

#### The Perfect Trade-off
Why not just track and retransmit each individual Code Block on its own? Because tracking dozens of Code Blocks independently would require massive feedback identifiers, creating too much control signaling overhead. Grouping them into **CBGs** strikes the perfect balance—drastically reducing retransmission sizes without cluttering the network with excessive tracking overhead.

## 📊 Summary of MAC Operations

The MAC layer is the ultimate scheduling coordinator and fast hardware link corrector:

![[MAC_Slide_10.png]]

* **Multiplexing:** Connects logical channels (services) to physical transport channels.
* **Control Elements (MAC CEs):** Injects low-overhead control plane adjustments (Timing Advance, BSR).
* **Carrier Aggregation:** Splits a single thread across multiple cell components.
* **Asynchronous HARQ & CBG:** Accelerates error repair with ultra-fine retransmission granularity to maximize airwave spectrum efficiency.
