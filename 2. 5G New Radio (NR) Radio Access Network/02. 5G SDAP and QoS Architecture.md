# 02. 5G SDAP and QoS Architecture

The **Service Data Adaptation Protocol (SDAP)** is a brand-new Layer 2 protocol sublayer introduced by 3GPP in Release 15 exclusively for 5G New Radio (NR) user-plane traffic. Sits at the very top of the Layer 2 Radio Access Network (RAN) protocol stack, positioned directly below the application layer (IP data) and directly above the Packet Data Convergence Protocol (PDCP) layer, SDAP serves as the critical engine mapping internet-level traffic profiles to the physical radio link.

## đźŽŻ Why Was SDAP Invented? (The Architectural Shift in QoS)

To understand SDAP, you must understand how 5G handles Quality of Service (QoS) differently than legacy generations (such as 4G LTE).

### A. Diverse Application Demands
5G New Radio is designed to simultaneously support a highly diverse mix of application classes from a single device (e.g., a connected autonomous vehicle running video streaming, telematics, radar sensing, and safety controls). These applications require vastly different over-the-air handling:

![[Materials/Slides/SDAP_Slide_2.png]]

* **Conversational Voice:** Low data rate, but requires highly continuous, low-latency processing.
* **Mobile Broadband (eMBB):** Ultra-high burst speeds but can tolerate moderate jitter.
* **Massive Machine-Type (mMTC/IoT):** High device density with lightweight transactional packets, demanding signaling efficiency over data rate.
* **Mission-Critical (URLLC):** Requires sub-millisecond latencies and a mathematically guaranteed $\ge 99.999\%$ transmission reliability.

If the network treated all of these incoming IP packets identically, it would be impossible to fulfill their contradicting performance metrics. Therefore, 5G manages data traffic using a brand-new end-to-end QoS architecture.

### B. The Problem: Core Flows vs. Radio Tracks
The architectural bottleneck lies in the mismatch between how the Core Network and the Radio Access Network classify and handle user traffic:

![[Materials/Slides/SDAP_Slide_3.png]]

1. **The Core Network View:** The 5G Core Network (specifically the User Plane Function or UPF) classifies and groups incoming application packets into abstract pipelines called **QoS Flows**. Every packet belonging to a specific QoS Flow is stamped with a unique 6-bit identifier called a **QFI (QoS Flow Identifier)**.
2. **The Access Network View:** The 5G Radio Access Network (the gNodeB base station) does not understand "flows." It can only transmit data across the airwaves using pre-configured, physical Layer 2 pipes called **DRBs (Data Radio Bearers)**.

Because there is a misalignment between the Core's QoS Flows and the RAN's Data Radio Bearers, an intermediary translator was required. **SDAP is that translator.** Its fundamental purpose is to map abstract Core QoS Flows onto the physical over-the-air Data Radio Bearers.

## đź› ď¸Ź QoS Flow Types and Performance Parameters

To guarantee the appropriate resource allocation, 5G classifies QoS Flows into three distinct operational categories, each governed by specialized metrics:

![[Materials/Slides/SDAP_Slide_4.png]]

### 1. GBR (Guaranteed Bit Rate)
Used for bandwidth-demanding, steady-state services (such as conversational voice [VoNR] or live video streams):
* **GFBR (Guaranteed Flow Bit Rate):** The absolute minimum bit rate mathematically guaranteed by the network.
* **MFBR (Maximum Flow Bit Rate):** The peak bit rate ceiling allowed for this specific flow.
* **Max Packet Loss Rate:** The maximum tolerable percentage of lost packets.

### 2. Non-GBR (Non-Guaranteed Bit Rate)
Used for bursty, best-effort services (such as web browsing, social media, or progressive video buffering):
* **Session-AMBR (Aggregate Maximum Bit Rate):** The maximum combined bit rate allowed across all active Non-GBR flows in a specific PDU Session.
* **UE-AMBR:** The maximum combined bit rate allowed across the entire device.

### 3. Delay-Critical GBR
Used for extreme real-time control loops (such as V2X cooperative collision avoidance, remote robotics, or grid automation):
* **MDBV (Maximum Data Burst Volume):** The maximum amount of data the PHY layer must transmit within a single delay budget window.
* Defines extremely low over-the-air latency budgets (e.g., 5 ms to 20 ms).

![[Materials/Slides/SDAP_Slide_5.png]]

## đź› ď¸Ź The 2 Core Functions of SDAP

According to 3GPP TS 37.324 standards, a separate instance of the SDAP protocol is configured and executed for every single active **PDU Session** (Packet Data Network Session) on the User Equipment (UE). SDAP executes two primary mandates:

### 1. Transfer of User Plane Data
Acts as the interface receiving raw IP packets (SDAP SDUs) from the application layer, executing protocol encapsulation, and forwarding them down through the radio stack (PDCP $\rightarrow$ RLC $\rightarrow$ MAC $\rightarrow$ PHY) to be converted into physical radio waves.

### 2. Mapping Between QoS Flows and Data Radio Bearers (DRBs)
SDAP decides exactly which over-the-air channel a packet must take. It does this via two architectural directions:

#### Downlink Multiplexing (Network âž” Phone)
When data flows down from the internet:
1. The **UPF (User Plane Function)** in the Core network classifies the data stream and marks the packets with a **QFI**.
2. The packets travel through an **N3 GTP-U tunnel** down to the gNodeB base station.
3. When the packets hit the gNodeB, the **SDAP layer reads the QFI** and maps the traffic to the corresponding pre-configured Data Radio Bearer (DRB) over the air.

![[Materials/Slides/SDAP_Slide_7.png]]

* **Flow Multiplexing (Aggregation):** SDAP does not require a 1-to-1 mapping. It can multiplex multiple QoS Flows into a *single* Data Radio Bearer. For example, if you are running a Zoom video call and a WhatsApp video call simultaneously, the UPF might isolate them into separate QoS Flows. However, because both are conversational video apps requiring identical delay-critical treatment (GBR QFI 2), the gNodeB's SDAP layer can bundle (multiplex) both flows together, sending them over a single high-priority Data Radio Bearer (e.g., DRB 2).

#### Uplink Mapping (Phone âž” Network)
When the smartphone needs to transmit data back up to the cell tower, its own internal client-side SDAP layer must decide which DRB to utilize. 3GPP defines two strict mechanisms for tracking uplink QoS:

![[Materials/Slides/SDAP_Slide_8.png]]

1. **Explicit Mapping:** The gNodeB base station pre-configures the phone's ruleset by sending a **Radio Resource Control (RRC)** control plane signaling message. This message explicitly instructs the phone's SDAP layer: *"Any uplink packet matching QFI X must be placed into Data Radio Bearer Y."* The phone stores this in its lookup table and executes it blindly.
2. **Reflective Mapping:** This is a highly advanced, ultra-efficient feature exclusive to 5G. Instead of wasting valuable control plane signaling overhead constantly sending explicit updates, the phoneâ€™s client-side SDAP layer actively **monitors incoming downlink traffic**. If the phone notices that the network is currently delivering a specific QoS Flow over Data Radio Bearer 3, the phone's SDAP layer instantly "reflects" that behavior. It automatically updates its internal routing table to use Data Radio Bearer 3 for all corresponding *uplink* packets traveling back to the tower.

## đź“¦ Detailed Anatomy of an SDAP Packet (SDU vs. PDU)

To see exactly how this protocol executes at a bit-level, let's look at its encapsulation mechanics.

1. **SDAP SDU (Service Data Unit):** This is the raw upper-layer packet (such as a Google Weather App update or an IP data packet) as it arrives at the top of the SDAP boundary.
2. **SDAP PDU (Protocol Data Unit):** SDAP processes the SDU by prepending a tiny, highly optimized **1-byte (8-bit) control header** directly to the front of the payload.

$$\text{IP Packet (SDAP SDU)} \longrightarrow \text{[SDAP Header (1 byte)] + [IP Packet]} \longrightarrow \text{SDAP PDU (passed to PDCP)}$$

### The 1-Byte SDAP Header Bit-Map

An SDAP header byte is structured into three explicit bit-fields:

```text
+---+---+---+---+---+---+---+---+
| R | D |     QFI (6 bits)      |
+---+---+---+---+---+---+---+---+
| 7 | 6 | 5 | 4 | 3 | 2 | 1 | 0 |  <- Bit Positions
```

* **QFI (6 bits â€” Bits 0 to 5):** This field explicitly carries the **QoS Flow Identifier**. Because it uses 6 bits, a single PDU session can dynamically scale to support up to $2^6 = 64$ distinct, concurrent QoS flows ($0\text{ to }63$).
* **RDI (1 bit â€” Bit 6):** **Reflective QoS Flow to DRB Indicator**. This bit tells the smartphone whether it must activate or refresh its reflective mapping rules for this specific flow.
* **RQI (1 bit â€” Bit 7):** **Reflective QoS Indicator**. This bit signals to the phone whether reflective QoS tracking must be passed up and applied to the non-radio Core parameters (the Non-Access Stratum / NAS layer).

Once the SDAP layer finishes stamping this 1-byte control header onto the payload, the completed **SDAP PDU** is handed down to the next sublayer (**PDCP**), where it is treated as a fresh **PDCP SDU** to undergo header compression via RoHC, ciphering, and over-the-air encryption.

### đź’ˇ Granular Technical Nuance: Header-less SDAP PDUs
To maximize over-the-air efficiency for simple bearers where dynamic QoS routing is not required, 3GPP allows the SDAP layer to be configured in a **Header-less** mode.
* When configured via RRC signaling, SDAP does not add the 1-byte header to packets on that specific Data Radio Bearer (DRB).
* This eliminates the 1-byte overhead completely, sending raw SDAP SDUs straight through as SDAP PDUs. This is particularly valuable for lightweight control messaging or highly static IoT links.

## đź”— Concrete Mapping Example

Inside a single PDU Session, a user may run three applications:
* **Web Browsing:** Best-effort (Non-GBR QFI 9) $\rightarrow$ Mapped by SDAP to **Radio Bearer 1**.
* **YouTube Video:** Progressive video (Non-GBR QFI 9) $\rightarrow$ **Multiplexed** by SDAP to **Radio Bearer 1**.
* **Zoom Video & WhatsApp Video:** Conversational video (GBR QFI 2) $\rightarrow$ Both are **Multiplexed** by SDAP to **Radio Bearer 2**.
