# 03. 5G PDCP Sublayer Architecture

The **Packet Data Convergence Protocol (PDCP)** is a highly specialized Layer 2 radio access network sublayer. In the User Plane, it sits directly underneath the **SDAP** layer, processing incoming SDAP PDUs as its own **PDCP SDUs**. In the Control Plane, it sits directly beneath the **RRC** layer. For both planes, its processed output (**PDCP PDUs**) is handed straight down to the **RLC** layer.

![[PDCP_Slide_1.png]]

---

## 🗺️ The Four Foundational Pillars of PDCP

According to 3GPP standards (TS 38.323), PDCP governs four core system requirements to ensure security, routing robustness, and transmission efficiency over volatile wireless channels:

![[PDCP_Slide_2.png]]

1. **Header Compression (RoHC Engine)**
2. **Ciphering and Integrity Protection**
3. **Routing and Duplication of Split Bearers**
4. **In-Sequence Delivery, Retransmission, and Reordering**

---

### 🔥 Pillar 1: Header Compression (The RoHC Engine)

Internet protocols are inherently verbose. When an application generates traffic, it wraps the raw data payload in large network routing headers:

* **IPv4 Headers:** 20 bytes standard IP + 20 bytes TCP = 40 bytes.
* **IPv6 Headers:** 40 bytes standard IP + 8 bytes UDP + 12 bytes RTP = 60 bytes.

#### The Radio Problem
For lightweight, small-payload applications—such as Voice over 5G (VoNR) or brief acknowledgement (ACK) packets where the payload itself might only be 20 bytes—the internet routing header itself can match or completely dwarf the size of the actual data payload. Transmitting 40 to 60 bytes of repetitive header information over the air with every tiny packet is a massive waste of precious radio spectrum, crippling capacity.

![[PDCP_Slide_3.png]]

#### The PDCP Solution
PDCP drastically shrinks this overhead by utilizing the **RoHC (Robust Header Compression)** protocol framework:

* **On the Transmitter Side:** The RoHC engine analyzes the stream of IP headers, strips out fixed fields that do not change from packet to packet (such as IP addresses, version numbers, and port configurations), and compresses the massive 40-to-60 byte header down to a tiny **2-to-3 byte compressed marker** before wireless transmission.
* **On the Receiver Side:** The receiving entity's PDCP layer feeds the compressed block into its own RoHC decompressor. It perfectly reconstructs the original full-length IPv4/IPv6 header structure before handing it over to the wired network.

### 🔐 Pillar 2: Ciphering and Integrity Protection

Security operations occur directly within the PDCP layer to shield the user and the network from external attacks, ensuring confidentiality and authenticity:

![[PDCP_Slide_4.png]]

* **Ciphering (Encryption):** This process prevents over-the-air eavesdropping. PDCP mathematically encrypts both user plane data and control plane signaling messages, ensuring that anyone sniffing the raw radio spectrum sees only unreadable ciphertext.
* **Integrity Protection:** This ensures **Data Origin Authentication** and prevents payload tampering. It appends a cryptographic signature to the packet to prove it originated from the legitimate user or network node, rather than an attacker attempting to inject rogue commands.

#### A. Explicit Cryptographic Key Separation
PDCP uses separate, distinct keys derived from the master anchor key ($K_{\text{gNB}}$) to isolate security contexts across different layers and data types:
* $K_{\text{RRC, enc}}$: Key used to encrypt/cipher RRC control plane messages.
* $K_{\text{RRC, int}}$: Key used to handle integrity protection on RRC control plane messages.
* $K_{\text{UP, enc}}$: Key used to encrypt/cipher User Plane application data.
* $K_{\text{UP, int}}$: Key used to handle integrity protection on User Plane application data (optional, used in high-security settings).
* $K_{\text{NAS, enc}}$ / $K_{\text{NAS, int}}$: Keys used for core network signaling (speaking directly to the AMF).

#### B. Granular Security Inputs/Parameters
The 5G security engine requires five explicit input parameters to generate a unique keystream, ensuring that identical payloads produce completely different ciphertext blocks to prevent replay attacks:

1. **Key:** 128-bit key ($K_{\text{RRC/UP/NAS, enc/int}}$).
2. **Count:** A 32-bit counter derived from the **PDCP Sequence Number (SN)** and the **Hyper Frame Number (HFN)**. HFN increments every time the PDCP SN wraps around. Count increases with each packet to guarantee uniqueness.
3. **Bearer:** 5-bit Bearer identity (DRB or SRB index), ensuring different radio bearers use different keystreams even with the same keys.
4. **Direction:** 1-bit value (0 for Uplink, 1 for Downlink), preventing keystream reuse between TX and RX.
5. **Length / Message:** The plaintext block (for ciphering) or the actual message payload (for integrity).

#### C. Cryptographic Engine Operations
* **Ciphering:** The inputs are fed into the **5G Encryption Algorithm (NEA)** to produce a *Keystream block*, which is then XORed with the plaintext block to output the *Ciphertext block*.
  $$\text{Plaintext} \oplus \text{Keystream (NEA)} \longrightarrow \text{Ciphertext}$$
* **Integrity:** The inputs are fed into the **5G Integrity Algorithm (NIA)** to produce a 32-bit **MAC-I (Message Authentication Code for Integrity)** or **NAS-MAC**, which is appended to the packet. The receiver calculates its own **XMAC-I / XNAS-MAC** and compares them; if they mismatch, the packet is silently discarded.

### 🔀 Pillar 3: Routing and Duplication of Split Bearers

In modern 5G deployments (especially Non-Standalone Dual Connectivity or MR-DC), a device doesn't talk to just one base station. It connects to a **Master Cell Group (MCG)** (LTE eNodeB or 5G gNodeB) and a **Secondary Cell Group (SCG)** (5G gNodeB) concurrently to aggregate bandwidth or guarantee link robustness.

![[PDCP_Slide_5.png]]

When a network path is configured as a **Split Bearer**, the single data stream coming down from the Core network hits the PDCP layer, which acts as the master junction box. PDCP has two ways to distribute these packets across the dual links:

1. **Routing:** PDCP dynamically splits the traffic based on capacity. It can send Packet 1 and 2 through the Master Cell's lower RLC/MAC stack, while sending Packet 3 and 4 across the backhaul interface (like $X2'$ or $Xn$) to be transmitted by the Secondary Cell.
2. **Duplication:** This is a key feature for ultra-reliable **URLLC** applications. Instead of splitting traffic, PDCP makes an exact carbon copy of a packet. It drops the original packet down the Master Cell route and fires the duplicate packet down the Secondary Cell route simultaneously. If wireless interference drops the packet on one cell, the other cell guarantees its safe arrival. The receiving PDCP safely discards the duplicate once the first copy arrives based on its Sequence Number (SN).

---

### ⏱️ Pillar 4: In-Sequence Delivery and Reordering

Because radio channels change constantly and split bearers send data across different paths, packets frequently arrive out of order at the receiver (e.g., Packet 1, Packet 2, Packet 4, then finally Packet 3).

To fix this, PDCP can be configured to operate in two modes, depending on what the application needs:

![[PDCP_Slide_6.png]]

#### 1. Out-of-Sequence Delivery Mode
For protocols like UDP or applications that have their own higher-layer reordering mechanisms (like certain TCP apps), PDCP can instantly flush packets up to the application layer the exact microsecond they arrive over the air, completely bypassing any internal queueing delay.

#### 2. In-Sequence Delivery Mode
For strict applications like video streaming or files that require absolute chronological accuracy, PDCP forces strict order:
* **The Mechanism:** On the transmitter side, PDCP stamps an explicit **Sequence Number (SN)** onto the header of every single packet.
* When the packets arrive out of order at the receiver, PDCP parks them inside a **reordering buffer** and starts a reordering timer (`t-Reordering`).
* If Packet 4 arrives before Packet 3, PDCP holds Packet 4 back in the buffer. It will not let the higher application layers see Packet 4 until the delayed Packet 3 safely arrives.
* Once Packet 3 lands, PDCP flushes them both up in perfect chronological order.
* If `t-Reordering` expires before Packet 3 arrives, PDCP assumes the packet was permanently lost, flushes Packet 4, and advances its window to prevent system stalling.

## 🛠️ The Functional Bypass Exception (Control vs. User Plane)

PDCP treats Control Plane and User Plane traffic slightly differently to keep the network running efficiently:

* **User Data Path:** Utilizes the full suite of PDCP operations—Header Compression (RoHC), Ciphering, Split Bearer Routing, and In-Sequence Sequence Number tracking.
* **Control Signaling Path (The Bypass):** Network signaling messages (like RRC configuration commands) do not require header compression because they don't use standard IP headers. Therefore, control plane messages **completely bypass the Header Compression engine**. However, they are strictly routed through the **Ciphering** and **Integrity Protection** blocks to maintain airtight security.

## ⚙️ Summary of the Transmitter and Receiver PDCP Architecture

The absolute data pipelines through the PDCP engine on both sides of the network are outlined as follows:

![[PDCP_Slide_7.png]]

### A. Transmitter Side (SDU ➔ PDU)
$$\text{SDAP PDU (PDCP SDU)} \longrightarrow \text{Header Compression (RoHC)} \longrightarrow \text{Integrity Protection (MAC-I)} \longrightarrow \text{Ciphering (NEA)} \longrightarrow \text{Add PDCP SN Header} \longrightarrow \text{Routing / Duplication} \longrightarrow \text{PDCP PDU to RLC}$$

### B. Receiver Side (PDU ➔ SDU)
$$\text{PDCP PDU} \longrightarrow \text{Reordering Buffer \& Duplicate Discarding} \longrightarrow \text{Deciphering (NEA)} \longrightarrow \text{Integrity Verification (Compare MAC-I \& XMAC-I)} \longrightarrow \text{Header Decompression (RoHC)} \longrightarrow \text{Remove PDCP Header} \longrightarrow \text{SDAP PDU (PDCP SDU)}$$
